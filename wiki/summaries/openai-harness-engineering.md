# Harness Engineering: Leveraging Codex in an Agent-First World

**Source URL:** https://openai.com/index/harness-engineering/
**Date:** February 11, 2026
**Author:** Ryan Lopopolo (Member of the Technical Staff, OpenAI)

## Summary

OpenAI's engineering team ran a five-month experiment building an internal product with zero manually-written code—every line of application logic, tests, CI configuration, documentation, observability, and tooling was generated by Codex (GPT-5.3). The repository grew to ~1 million lines of code across 1,500 pull requests with a team of just 3–7 engineers, achieving roughly 3.5 PRs per engineer per day and estimated 10× time savings versus manual coding. The experiment formalized "harness engineering" as the practice where engineers design environments, specify intent, and build feedback loops rather than writing code, redefining the software engineer's role from code author to system architect who enables agents to do reliable work.

## Key Claims

- **Zero manually-written code:** Over five months, OpenAI shipped an internal product with 0 lines of human-written code—all application logic, tests, CI configuration, documentation, observability, and internal tooling generated by Codex (GPT-5.3)
- **1M lines of code:** Repository grew to ~1 million lines of code across application logic, infrastructure, tooling, documentation, and internal developer utilities
- **1,500 pull requests:** Roughly 1,500 PRs opened and merged over five months with a small team of 3 engineers initially, growing to 7 engineers
- **3.5 PRs per engineer per day:** Average throughput that increased as the team grew
- **10× time savings:** OpenAI estimates the product was built in ~1/10th the time it would have taken to write code by hand
- **August 2025 start:** First commit landed in late August 2025 with repository structure, CI, formatting rules, package manager setup, and application framework all generated by Codex CLI using GPT-5
- **AGENTS.md as table of contents:** OpenAI tried "one big AGENTS.md" approach and it failed (context crowding, guidance overload, rapid rot, hard to verify); shifted to short ~100-line AGENTS.md as map pointing to structured docs/ directory as system of record
- **Progressive disclosure architecture:** Repository knowledge base structured as docs/ with design-docs/, exec-plans/ (active/completed/tech-debt), generated/, product-specs/, references/ subdirectories; enables agents to start with small stable entry point and navigate to deeper context
- **Mechanical enforcement:** Dedicated linters and CI jobs validate knowledge base is up-to-date, cross-linked, and structured correctly; recurring "doc-gardening" agent scans for stale docs and opens fix-up PRs
- **Agent legibility as first principle:** Codebase optimized for Codex's legibility; anything not in-repo (Google Docs, Slack, people's heads) "effectively doesn't exist" to the agent; repository-local versioned artifacts (code, markdown, schemas, executable plans) are all it can see
- **Layered architecture enforced mechanically:** Each business domain divided into fixed layers (Types → Config → Repo → Service → Runtime → UI) with strictly validated dependency directions; cross-cutting concerns (auth, connectors, telemetry, feature flags) enter via single explicit interface (Providers); enforced via custom linters and structural tests
- **Ralph Wiggum Loop:** Codex reviews its own changes locally, requests additional agent reviews (local + cloud), responds to feedback, iterates until all agent reviewers satisfied; humans may review PRs but aren't required; review effort shifted almost entirely to agent-to-agent
- **Application legibility for QA:** Made app bootable per git worktree so Codex can launch one instance per change; wired Chrome DevTools Protocol into agent runtime for DOM snapshots, screenshots, navigation; Codex can reproduce bugs, validate fixes, reason about UI behavior directly
- **Observability legibility:** Logs, metrics, traces exposed to Codex via local ephemeral observability stack per worktree; agents query logs with LogQL and metrics with PromQL; enables prompts like "ensure service startup completes in under 800ms"
- **Six-hour autonomous runs:** Team regularly sees single Codex runs work on a single task for upwards of six hours (often while humans sleeping)
- **End-to-end autonomy:** Codex can now validate codebase state, reproduce bug, record video of failure, implement fix, validate fix, record resolution video, open PR, respond to feedback, detect/remediate build failures, escalate only when judgment required, merge change
- **Garbage collection for AI slop:** Team initially spent every Friday (20% of week) manually cleaning up "AI slop"; solution: encode "golden principles" directly into repo with recurring cleanup process—background Codex tasks scan for deviations, update quality grades, open targeted refactoring PRs (most automerged)
- **Throughput changes merge philosophy:** Repository operates with minimal blocking merge gates; PRs short-lived; test flakes addressed with follow-up runs rather than blocking indefinitely; "corrections are cheap, waiting is expensive"
- **Boring tech for agent legibility:** Favored dependencies/abstractions that can be fully internalized and reasoned about in-repo; "boring" technologies easier for agents due to composability, API stability, representation in training set; sometimes cheaper to have agent reimplement subsets than work around opaque upstream behavior (example: custom map-with-concurrency helper with OpenTelemetry instrumentation and 100% test coverage instead of generic p-limit package)
- **Taste invariants as linter error messages:** Statically enforce structured logging, naming conventions, schema/type patterns, file size limits, platform-specific reliability requirements via custom lints; write error messages to inject remediation instructions into agent context
- **Codex uses standard dev tools directly:** Agents use gh, local scripts, repository-embedded skills to gather context without humans copying/pasting; pull review feedback, respond inline, push updates, often squash and merge their own PRs
- **Real users:** Product has internal daily users and external alpha testers; has been used by hundreds of users internally including daily power users
